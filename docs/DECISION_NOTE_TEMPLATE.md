# 결정 노트 템플릿

> 모든 페이즈/서브페이즈 진입 시 `docs/research/<phase>-decision.md` 또는 `<phase>-reinforcement.md`로 작성.
> 본 템플릿은 **negative space 보존 + 다음 세션 컨텍스트 인계**를 목적으로 한다 — 결정의 결론뿐 아니라 *왜 다른 선택지를 거부했는지*를 의무적으로 남긴다.
> 6 섹션 모두 필수 — "해당 없음"이면 한 줄이라도 명시. 빈 섹션 금지.

## 작성 가이드

### "기각안 + 이유"가 가장 중요한 이유

세션 compaction은 *채택 결과*는 잘 살리지만 *거부한 대안*은 빠르게 옅어진다. 다음 세션이:

1. 이미 검토하고 거부한 라이브러리를 *처음부터 검토* → 같은 결론 다시 도달 (시간 낭비).
2. 거부한 안의 단점을 모르고 *재선택* → 같은 함정에 다시 빠짐 (퀄리티 사고).

이를 한 줄로 막는 게 §2이다. "X를 봤지만 Y 때문에 거부 — Z가 바뀌면 재검토" 한 문장이 1시간을 살린다.

### 테스트 invariant ↔ negative space

§4의 invariant 테스트는 §2의 "기각안"이 *코드 차원에서* 다시 채택되는 걸 막는다. 예를 들어:

- §2에 "non-deterministic recommender 거부 — 같은 PC 다른 추천이면 신뢰 하락"
- §4에 `determinism_invariant` 테스트 100회 반복

이 둘이 짝을 이루면 다음 세션이 randomness를 우연히 도입해도 즉시 빨간불.

### 결정 노트와 메모리의 분리

- **결정 노트** = 결정의 *왜* + *기각안* + *테스트 invariant* (영속, repo).
- **메모리** = 다음 세션 시작 시 즉시 알아야 할 *메타 맥락* (예: "현재 Phase 2'.c.2 진행 중, 다음 standby는 Phase 3'").

메모리에 결정의 *왜*를 적지 말 것 — 그건 결정 노트의 책임. 메모리는 짧고 자주 갱신되는 항목만.

## 기존 결정 노트 retrofit 가이드

기존 노트(예: `phase-2pa-catalog-decision.md`)에 §2 "기각안" 섹션이 약하면 다음 sub-phase 진입 시 1단계로 *기각안 회상*을 추가한다. 완벽한 retrofit이 어렵다면 **새 결정 노트부터 본 템플릿 적용**.
