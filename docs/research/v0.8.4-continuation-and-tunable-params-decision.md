## 0. 결정 요약 (8가지)

1. **A1 — finish_reason 채널 신설**: `ChatEvent::Completed { took_ms, finish_reason: FinishReason }` 확장 (와이어 추가, 백워드 호환). Ollama `done_reason="length"` / llama.cpp `finish_reason="length"` / `stop_type="limit"` → 공통 `FinishReason::Length` 매핑.
2. **A2 — 자동 continuation은 페르소나 응답이 아닌 *리포트 합성*에만**: 페르소나 응답은 본질적으로 짧음(질문 1개당 평균 50~300 토큰). 잘려도 의도된 컷 — `system_extra`로 길이 가이드 + `num_predict` 슬라이더로 사용자 통제. 자동 재시도 X. 잘림 발생 시 응답 옆에 "잘렸어요 · 토큰 한도를 늘려볼래요?" 인라인 칩만 표시.
3. **A3 — 리포트 프롬프트는 frontend chunked map-reduce**: 응답 N×M개를 client-side에서 토큰 추정 휴리스틱(한국어 1글자 ≈ 2.5 토큰)으로 청크 분할 → 각 청크는 "이건 X부터 Y까지 응답이에요. 다른 chunk가 더 있어요" header가 붙은 *부분 분석 프롬프트* + 마지막 1개의 *종합 합성 프롬프트* 생성. 백엔드는 텍스트 빌드만, LLM 호출은 외부 (ChatGPT/Claude/Gemini)에서 사용자가 직접 — 기존 v0.8.3 클립보드 흐름 유지.
4. **A4 — `SamplingParams` struct는 `chat-protocol` crate에**: 두 어댑터가 이미 `chat_protocol::ChatMessage/ChatEvent/ChatOutcome`을 공유하므로 동일 위치. `Option<SamplingParams>`로 추가, `chat_stream` 시그니처에 5번째 인자 추가 — 모든 호출처(personas/survey, chat IPC, lmstudio adapter)에서 `None` 전달로 1차 컴파일 OK 후 점진 채택.
5. **A5 — 사용자 노출 5종 + 프리셋 3종**: `max_tokens` / `temperature` / `top_p` / `repeat_penalty` / `seed`. Lite/Balanced/Creative 프리셋 + "직접 조절"은 details 펼침. top_k / num_ctx / mirostat / num_keep 등은 v0.8.5+ 후순위.
6. **A6 — 한국어 라벨 + slider/number input 매트릭스**: `max_tokens`=number(64~8192), `temperature`=slider(0.0~1.5), `top_p`=slider(0.1~1.0), `repeat_penalty`=slider(1.0~1.5), `seed`=number+"랜덤" 토글. 각각 helper text + hint chip ("값이 클수록 응답이 길어요" 등).
7. **A7 — 두 어댑터 wire mapping은 어댑터 내부**: `SamplingParams → Ollama options` / `SamplingParams → OpenAI request fields`. lmstudio는 OpenAI 호환이라 llama-cpp와 동일 경로. 어댑터 외부로 매핑 누수 X.
8. **A8 — 사용자 설정 영속**: `localStorage` "personas-survey-sampling" 키. 백엔드 settings.json까지 끌어들이지 않음 — Step 4 화면 한정 ephemeral state.

## 1. 채택안

### A1 — `FinishReason` 채널 신설 (chat-protocol)

**스키마 추가** (`crates/chat-protocol/src/lib.rs`):

```rust
#[derive(Debug, Clone, Copy, PartialEq, Eq, Serialize, Deserialize)]
#[serde(rename_all = "kebab-case")]
pub enum FinishReason {
    /// 정상 종료 (EOS / stop word / 자연 마침).
    Stop,
    /// 토큰 한계 도달 — 자동 continuation 또는 사용자 안내 트리거.
    Length,
    /// 사용자 cancel 또는 외부 abort (graceful early disconnect 등).
    Aborted,
    /// 모델 로드/언로드 등 메타 이벤트 (Ollama "load"/"unload"). 일반 응답 아님.
    Meta,
    /// 알 수 없음 (필드 누락 / 신규 값) — 안전 디폴트로 Stop과 동일 처리.
    Unknown,
}

#[derive(Debug, Clone, Serialize, Deserialize, PartialEq)]
#[serde(tag = "kind", rename_all = "kebab-case")]
pub enum ChatEvent {
    Delta { text: String },
    Completed {
        took_ms: u64,
        /// v0.8.4 추가 — 잘림 감지용. 백워드 호환은 `#[serde(default)]`로 보장.
        #[serde(default = "default_finish_reason")]
        finish_reason: FinishReason,
    },
    Cancelled,
    Failed { message: String },
}

fn default_finish_reason() -> FinishReason { FinishReason::Unknown }
```

**Ollama 매핑** (`adapter-ollama::chat_stream`):
- `ChatChunk`에 `done_reason: Option<String>` 추가.
- `done=true` chunk에서 `done_reason`을 보고 `"stop"→Stop`, `"length"→Length`, `"load"|"unload"→Meta`, 그 외→`Unknown`.
- graceful early disconnect 분기(EOF without `done=true`)는 `Aborted`로 표시.

**llama.cpp 매핑** (`adapter-llama-cpp::chat_stream`):
- `OpenAIChatChoice.finish_reason` (`Option<String>`)을 그대로 활용. `"stop"→Stop`, `"length"→Length`, 그 외→`Unknown`.
- `[DONE]` 마커 후 finish_reason 못 받았으면 `Unknown` (대부분 실제론 stop).

**테스트** (Phase 의무 invariant):
```rust
// crates/chat-protocol/src/lib.rs
#[test] fn finish_reason_round_trip_kebab() { /* Stop/Length/Aborted/Meta/Unknown */ }
#[test] fn legacy_completed_without_finish_reason_deserializes_unknown() { /* 백워드 */ }

// crates/adapter-ollama
#[test] fn chat_stream_completed_emits_length_when_done_reason_length() { ... }
#[test] fn chat_stream_completed_emits_stop_when_done_reason_stop() { ... }

// crates/adapter-llama-cpp
#[test] fn chat_stream_completed_emits_length_when_finish_reason_length() { ... }
```

**왜 `Aborted` 변종을 따로 두나**: graceful early disconnect(`delta_emitted` 분기)에서 사용자에게 "잘렸어요" 알림이 잘못 뜨면 안 됨. `Aborted`는 *transport 끊김*이지 *모델이 토큰 한계 친 게 아님* — UI 카피가 달라야 함.

### A2 — 페르소나 응답: 자동 continuation 거부, 사용자 통제로 회수

**핵심 사고**: 페르소나 한 명의 한 질문 응답은 평균 50~300 토큰. `num_predict` 디폴트가 `-1`(무한)인 환경에선 잘릴 일 자체가 거의 없음. 사용자 캡처에서 본 "## 현재 진행 중인 상황..." 잘림은 외부 LLM 리포트 합성 단계 — 거기에 자동 continuation을 적용해도 *외부 LLM*이라 우리가 조건 못 잡음. **페르소나 단계는 사용자가 max_tokens를 직접 조절하게 + 응답이 잘리면 인라인 알림**으로 충분.

**UI 동선** (PersonaSurvey Step 4):
- 응답 카드 위에 `FinishReason::Length`면 작은 칩 "응답이 잘렸어요 · 최대 토큰 늘려서 다시 받아볼래요?" 표시.
- 칩 클릭 시 sampling drawer 열고 `max_tokens` 입력에 자동 포커스.
- 자동 재호출 X — 사용자가 "다시 시작" 버튼 누르도록.

**왜 자동 재호출이 아닌가**: 자동 continuation은 (1) N×M개 페르소나 응답이라 batch 비용 폭발, (2) "이어서" 요청이 페르소나 narrative consistency를 깰 가능성, (3) 사용자가 "왜 갑자기 두 번 호출됐지" 혼란. 사용자 명시 트리거가 정직.

**대신 system_extra 가이드**: 기본 system prompt에 "300자 이내로 평소 말투로 답해 주세요" 같은 길이 힌트를 사용자가 자유롭게 추가할 수 있게 (이미 v0.8.3에 textarea 있음 — 디폴트 placeholder만 보강).

### A3 — 리포트 프롬프트: frontend map-reduce chunking

**현재 (v0.8.3)**: `personas_generate_report_prompt`가 question_summaries 전체를 단일 markdown으로 빌드 → 사용자가 클립보드 복사 → 외부 LLM 1회 paste. **문제**: 페르소나 100명 × 질문 10개 + 주관식 응답 샘플 5개씩 풀 인용 시 한국어 기준 50K~150K 토큰. ChatGPT-4o context 128K, Gemini 2M, Claude 200K. 잘릴 가능성 충분.

**채택**: `personas_generate_report_prompt`가 단일 String 대신 `ReportPromptPlan` 반환:

```rust
#[derive(Debug, Clone, Serialize)]
pub struct ReportPromptPlan {
    /// 청크가 1개면 single-shot — 기존 흐름 호환 (사용자 1번 paste).
    /// 2개 이상이면 chunked: 각 chunk를 순서대로 paste → 마지막에 합성 prompt paste.
    pub chunks: Vec<ReportPromptChunk>,
    /// 모든 chunk paste 후 사용자가 보낼 종합 합성 prompt. chunks.len() == 1이면 None.
    pub final_synthesis: Option<String>,
    /// UI 표시용 — "청크 N개 · 예상 토큰 약 X" 카피.
    pub estimated_tokens_total: u64,
}

#[derive(Debug, Clone, Serialize)]
pub struct ReportPromptChunk {
    pub seq: usize,           // 1-based
    pub total: usize,
    pub prompt: String,       // header + 부분 데이터 + 부분 분석 지시
    pub estimated_tokens: u64,
}
```

**chunking 알고리즘** (Rust 백엔드, decision: backend-side로 두는 게 frontend toxic dependency 줄임):
1. 한국어 토큰 추정: `text.chars().count() * 25 / 10` (1글자 ≈ 2.5 토큰 — `tiktoken-rs` cl100k_base 실측 한국어 평균. 보수적 상한.)
2. 사용자 선택 외부 LLM의 max input 토큰 보수 한계 (디폴트 80K — 거의 모든 모델이 안전).
3. 질문(`QuestionSummary`) 단위로 grouping — 한 질문은 한 chunk 안에 통째로. open_samples는 이미 5개로 제한돼 있고, option_counts는 보통 짧음.
4. 누적 토큰이 한계 80% 초과 시 새 chunk 시작.
5. 모든 chunk에 공통 header: `## 부분 분석 요청 — N분의 K청크`.
6. final_synthesis prompt: "위 N개 청크의 부분 분석을 종합해 ${ReportStyle} 양식으로 작성해 주세요" + 페르소나 분포 + 스타일 directive.

**왜 LangChain `MapReduceDocumentsChain` 패턴 채택 (Refine 기각)**:
- **Map-reduce**: 청크별 독립 분석 → 병렬 가능, 최종 합성 1회. *부분 분석은 사용자가 외부 LLM에서 paste-순차 호출하지만 의미 의존성은 X*. — 우리 시나리오에 정확히 맞음.
- **Refine**: 청크 간 의존 (이전 답변을 다음 청크 입력에 포함) — 사용자가 매번 결과를 복사해서 다음 chunk에 paste해야 함. UX 마찰 큼. *기각*.
- **TreeSummarize** (LlamaIndex): 다단계 트리 — 사용자에게 너무 많은 step. 페르소나 100명 정도까지는 2~4개 chunk면 충분 → tree depth 1로 map-reduce가 동등. *오버엔지니어링으로 기각*.

**UI 변경** (Step 4 RunReportCard):
- 기존 textarea 단일 → "청크 N개" stepper. 각 청크별 ClipboardCopy 버튼 + "복사 완료" 카운터.
- chunks.len() == 1이면 기존 UI 유지 (regression 방지).
- 청크 paste 순서를 잊지 않게 progress chip ("1/3 복사 완료 · 2/3 다음").

### A4 — `SamplingParams` struct 위치와 시그니처 확장

**파일**: `crates/chat-protocol/src/lib.rs` 신규 추가.

```rust
/// 사용자 조절 가능한 추론 파라미터 (v0.8.4).
///
/// 정책:
/// - 모든 필드 `Option` — None은 어댑터/모델 디폴트 사용.
/// - 어댑터별 wire 매핑은 각 어댑터 내부 (Ollama options vs OpenAI request fields).
/// - 와이어 호환은 `serde(skip_serializing_if = "Option::is_none")`로 보존.
#[derive(Debug, Clone, Default, Serialize, Deserialize, PartialEq)]
pub struct SamplingParams {
    /// 최대 응답 토큰 수. None이면 모델/서버 디폴트 (Ollama -1 = unlimited, llama.cpp -1 = infinity).
    #[serde(skip_serializing_if = "Option::is_none")]
    pub max_tokens: Option<u32>,
    /// 0.0 (deterministic) ~ 1.5+ (creative). None이면 어댑터 디폴트 (대개 0.7~0.8).
    #[serde(skip_serializing_if = "Option::is_none")]
    pub temperature: Option<f32>,
    /// nucleus sampling. 0.0~1.0.
    #[serde(skip_serializing_if = "Option::is_none")]
    pub top_p: Option<f32>,
    /// 1.0이면 페널티 없음. 1.1~1.3이 일반적.
    #[serde(skip_serializing_if = "Option::is_none")]
    pub repeat_penalty: Option<f32>,
    /// 재현성용. None이면 서버 랜덤 (Ollama -1, llama.cpp -1).
    #[serde(skip_serializing_if = "Option::is_none")]
    pub seed: Option<i64>,
}
```

**어댑터 시그니처 확장 (호환성 결정)**:
- 신규 메서드 추가 X. 기존 `chat_stream`에 `params: Option<&SamplingParams>` 5번째 인자 추가.
- 호출처는 모두 `None` 전달로 1차 컴파일 통과 → 페르소나/설문 호출처 1군데만 `Some(&params)` 채택.
- *왜 `Option<>`인가*: lmstudio adapter 등 다른 호출처를 한 번에 다 손대지 않기 위한 incremental migration. Phase 8'.x adapter-lmstudio도 동일 시그니처로 확장(별 sub-phase) 후 ChatPage IPC 단위 채택.

**리스크**: 시그니처 5번째 인자는 호출 가독성 약함 — 통합 args struct(`ChatStreamArgs`)로 가는 게 우아하지만, *지금 페르소나 1군데만 쓰는 단계에선 과한 리팩토링*. v0.8.5+ adapter-lmstudio 채택 시점에 args struct 검토.

**프리셋**:
- **정확하게 (Lite)**: temp=0.2, top_p=0.9, repeat_penalty=1.05. 객관식/척도 응답에 권장 — 사용자가 "객관식이 자꾸 자기 말 덧붙여요" 호소 시 권유.
- **균형 (Balanced)**: temp=0.7, top_p=0.95, repeat_penalty=1.1. 디폴트.
- **창의적 (Creative)**: temp=1.0, top_p=0.98, repeat_penalty=1.15. 주관식 깊이 + 다양성.

**왜 LM Studio처럼 raw slider 단독이 아닌가**: 한국 사용자(엔지니어 아닌 마케터/UX 리서처)가 1차 타깃. "temperature 0.7이 무슨 뜻이지?" 진입 장벽이 높음. 프리셋 선택 → "직접 조절" 펼침 패턴이 GPT4All / Open WebUI / LobeChat의 합의된 진입 패턴.

**왜 OpenWebUI처럼 "모든 모델에 영향" 글로벌 디폴트는 X**: 페르소나 시뮬은 Step 4 한정 ephemeral context. ChatPage(`/chat`)는 별 sub-phase에서 동일 패턴 채택할 때 함께 글로벌 디폴트 검토.

### A6 — 한국어 라벨 + slider/number input 매트릭스

**컴포넌트** (`apps/desktop/src/pages/persona-survey/SamplingDrawer.tsx` 신규):
- Drawer (오른쪽 슬라이드) — `prefers-reduced-motion` 토큰 자동 비활성.
- `role="dialog" aria-modal="true" aria-labelledby` 3종 세트.
- 첫 input (`max_tokens` number)에 auto-focus.
- Esc / 배경 클릭 닫기.
- 프리셋 3종을 `role="radiogroup"`로 묶고 자식 `role="radio" aria-checked`.

**입력 타입 결정**:
- **number input** (`max_tokens`, `seed`): 정수, 큰 범위, 사용자가 정확 값 입력 원할 가능성 높음.

## 2. 기각안 + 이유 (Negative space — 의무 섹션)

### 2.1 페르소나 응답 자동 continuation 채택 거부

- **검토**: ChatGPT 스타일 — finish_reason=length 감지 후 "계속해" user turn append + 동일 모델 재호출. LibreChat issue #3465 패턴.
- **거부 이유**:
  1. **페르소나 응답은 짧음**. 50~300 토큰 평균. `num_predict` 디폴트가 -1(무한)이면 잘릴 일 자체가 거의 없음 — 자동 continuation 코드는 99% deadweight.
  2. **Batch 비용 폭발 위험**. N(100명) × M(10문항) × 평균 1.5라운드 = 1500 호출. 사용자 PC RTX 2060(6GB)에서 30분 → 75분으로 늘어남. UX 후퇴.
  3. **Persona narrative consistency 깨짐**. "이어서 답해 주세요" prompt가 페르소나 일인칭 시점을 외부 메타시점으로 끌어당김. 응답 품질 저하.
  4. **사용자 통제권**. 잘림은 의도된 컷일 수도 있음 (system prompt가 "한 줄로 답해 주세요"). 시스템이 임의 재호출하면 사용자 의도 invalidate.
- **재검토 트리거**: 평균 응답 길이가 토큰 한계를 자주 친다는 사용자 피드백 (예: 주관식 비율이 매우 높은 설문 시뮬에서 잦은 컷). 그땐 *opt-in 토글*("길어지면 자동으로 이어 받을게요") + 최대 2라운드 제한.

### 2.2 외부 LLM 자동 호출 (OpenAI API key 직접 호출) 거부

- **검토**: 리포트 합성 단계를 외부 LLM에 자동 paste 대신 OpenAI/Anthropic API key를 사용자가 등록하면 LMmaster가 직접 호출. continuation도 자동 처리.
- **거부 이유**:
  1. **외부 통신 0 원칙 위배**. CLAUDE.md §4.2 — "외부 통신 0: localhost-only 바인딩, no-proxy". v1 절대 원칙. 외부 API key 흐름은 별도 architectural 결정 (ADR 신설 + UX 마법사 + key 보관 보안 모델 + 비용 표시 등) 필요.
  2. **사용자 비용 책임**. ChatGPT Plus / Claude.ai를 이미 쓰고 있는 사용자는 paste 흐름이 *추가 비용 0*. API 호출은 토큰당 과금 — 사용자 명시 동의 + 비용 가시화 필요.
  3. **v0.8.4 스코프 폭발**. ADR + key UX + 비용 표시 + rate limit 처리 = 별 sub-phase. 본 v0.8.4는 *2시간 분량 목표*.
- **재검토 트리거**: v1.x에서 "Workbench 외부 LLM 게이트웨이" 페이즈 진입 시 (ADR 신설 + 키 보관 모델 + 비용 표시).

### 2.3 backend Rust 토큰 카운팅 (`tiktoken-rs` + `/tokenize` 엔드포인트) 채택 거부

- **검토**: chunk 분할 시 정확한 토큰 카운트가 필요. tiktoken-rs cl100k_base + llama-server `/tokenize` 호출.
- **거부 이유**:
  1. **외부 LLM은 우리가 토크나이저를 모름**. 사용자가 chunk 받아 ChatGPT에 paste — tiktoken은 GPT 시리즈, Gemini는 SentencePiece, Claude는 BPE. 정확 카운트가 *외부* LLM에서 의미 없음.
  2. **휴리스틱이 80% 충분**. "한국어 1글자 ≈ 2.5 토큰" 휴리스틱(보수적 상한)은 거의 모든 토크나이저에서 over-estimate (안전). under-cut 위험만 막으면 OK.
  3. **의존성 추가**. tiktoken-rs ≈ 4MB 바이너리. 페르소나 시뮬 1군데 위해 introduce 큼.
- **재검토 트리거**: v0.8.5+ "내부 LLM(Ollama/llama.cpp)으로 리포트 합성도 자동화" 페이즈. 그땐 llama-server `/tokenize` 정확 카운트 필요.

### 2.4 LangChain Refine 패턴 채택 거부

- **검토**: 청크 N개를 순차 처리, 각 청크의 답변을 다음 청크 입력에 누적. 정보 손실 가장 적음.
- **거부 이유**:
  1. **사용자 마찰 큼**. 사용자가 chunk 1 paste → 외부 LLM 답변 받음 → *그 답변을 복사해서* chunk 2 prompt에 paste. 4단계 chunk면 7번의 복사-paste.
  2. **순서 의존성**. 외부 LLM 응답이 다음 chunk에 들어가야 하는데, 우리는 그 응답을 캡처할 수단 X (외부 LLM ↔ LMmaster 양방향 X).
  3. **Map-reduce가 본질에 더 가까움**. 페르소나 응답은 *독립* 데이터 포인트 — 청크별 분석은 의미상 독립 가능. 마지막에 합성만 1회.
- **재검토 트리거**: 응답이 *시계열 대화*(체이닝된 후속 질문) 같은 의존 구조로 진화하면. 현재 설문은 독립 문항.

### 2.5 GUI에서 top_k / num_ctx 노출 거부 (v0.8.5+로 이월)

- **검토**: LM Studio가 노출하는 "표준 5종"에 포함.
- **거부 이유**:
  1. **top_k는 top_p와 효과 중복**. 둘 다 sampling diversity 조절. 사용자가 "둘이 뭐가 달라요?" 질문하면 답하기 어려움 (raw GPT 사용자도 이 둘 의미 헷갈림). nucleus(top_p)가 deeper 효과 → top_p만 노출이 90% 케이스 충분.
  2. **num_ctx는 VRAM 영향 큼**. 4096 → 32768 늘리면 KV cache 폭발 → OOM. 사용자 PC 모델별 max 자동 감지 + warning 필요. *별 sub-phase에서 카탈로그 metadata와 결합*해야 안전.
- **재검토 트리거**: 사용자가 "긴 대화 맥락이 잘려요" 호소 (현재 페르소나는 1턴이라 num_ctx 무관).

### 2.6 sampling을 백엔드 settings.json에 persist 거부 (현 시점)

- **검토**: settings 파일에 "personas_default_sampling" 저장 → 다른 페이지(Chat)에서도 공유.
- **거부 이유**:
  1. **현재 공유 욕구 X**. Chat과 페르소나 시뮬은 다른 멘탈 모델. Chat은 1:1 대화, 페르소나는 batch 시뮬.
  2. **Settings 화면 UX 추가 필요**. 별 sub-phase 단위.
- **재검토 트리거**: ChatPage v8.x 단위에서 sampling 도입 시 — 그때 "프로젝트 단위 default sampling" 도입 결정 검토.

### 2.7 프리셋 raw 슬라이더 단독 (LM Studio 패턴) 거부

- **검토**: LM Studio Advanced Configuration처럼 모든 슬라이더를 평면 배치.
- **거부 이유**:
  1. **타깃 사용자 (마케터/UX 리서처) 진입 장벽**. "temperature가 뭐죠?" 만남 즉시 close.
  2. **GPT4All, Open WebUI Models workspace, LobeChat 모두 프리셋 + 펼침 패턴**. 합의된 진입 UX.
- **재검토 트리거**: ChatPage 고급 사용자 트랙 별도 화면 분리 시.

## 4. 테스트 invariant (sub-phase가 깨면 안 되는 동작)

### 4.1 chat-protocol crate

- **finish_reason round-trip**: `Stop`, `Length`, `Aborted`, `Meta`, `Unknown` 5변종 모두 kebab-case JSON ↔ enum.
- **legacy `Completed` 백워드**: `{"kind":"completed","took_ms":1234}`에서 `finish_reason` 누락 → `Unknown` 디폴트.
- **SamplingParams empty serialize**: 모든 필드 None일 때 JSON `{}` (skip 적용). `Default` 또한 모든 None.
- **SamplingParams round-trip**: 5필드 모두 채운 케이스 + 부분 채운 케이스.

### 4.2 adapter-ollama / adapter-llama-cpp

- **Length 감지**: mock server가 `done_reason="length"` (Ollama) / `finish_reason="length"` (llama.cpp) 보내면 `ChatEvent::Completed { finish_reason: Length, .. }` emit.
- **Stop 감지**: 평범 done → `Stop`.
- **Aborted (graceful disconnect) 분기**: delta 1건 이상 + transport drop → `Completed { finish_reason: Aborted }` (기존 graceful 패턴 유지). `Length`로 잘못 표시되면 사용자 카피 오해 — 회귀 가드.
- **`options` 필드 wire**: `SamplingParams` 채워서 호출 시 Ollama request body에 `options.num_predict` 등 직렬화 / `None`일 땐 `options` 필드 자체 skip.
- **Determinism (참고)**: 동일 seed + 동일 prompt → 동일 응답 (서버 행동, 우리는 wire만 검증).

### 4.3 personas backend (chunking)

- **단일 chunk 케이스**: 응답이 80K 토큰 이하 → `chunks.len() == 1`, `final_synthesis == None`. 기존 v0.8.3 흐름과 wire 동등.
- **다중 chunk 케이스**: 100명 × 10문항 + 풀 인용 → `chunks.len() >= 2`. 각 chunk header에 "N/M" + 한국어 프리앰블 포함.
- **합성 prompt**: chunks.len() ≥ 2일 때 `final_synthesis`에 페르소나 분포 + ReportStyle directive + "위 N개 청크 종합" 지시 포함.
- **순서 보존**: question 단위로 grouping — 한 question이 두 chunk에 쪼개지지 않음.
- **토큰 추정 휴리스틱**: 한국어 텍스트 N글자 → ⌈N × 2.5⌉ 토큰. 영어/숫자만 있는 경우는 1글자 ≈ 0.4 토큰 (upper bound 유지하므로 conservative). 실측 케이스 1건 단언.

### 4.4 frontend (PersonaSurvey + SamplingDrawer)

- **a11y**: vitest-axe `violations.toEqual([])` for SamplingDrawer (role=dialog, aria-modal, focus trap).
- **프리셋 라디오**: `role="radiogroup"` + 자식 `role="radio" aria-checked` + 화살표 키 navigation.
- **Esc 닫기**: keyboard event → drawer close + focus restore.
- **localStorage 영속**: 프리셋 변경 → reload → 동일 프리셋 복원.
- **chunked 리포트 UI**: chunks.length > 1일 때 stepper "N/M 복사 완료" 정확 카운트.
- **단일 chunk 회귀**: chunks.length === 1일 때 기존 textarea + ClipboardCopy 단일 흐름 그대로 (regression guard).
- **한국어 카피 단언**: "응답이 잘렸어요" 인라인 칩 + "최대 답변 길이" SamplingDrawer 라벨 marker text 단언 (i18n 키 broken 회귀 방지).

### 4.5 i18n

- **ko/en 1:1**: `personas.sampling.*` 키 5종 (label + helper) + 프리셋 3종 = 13키 양쪽 완전 동기화.
- **fallback 깨짐 없음**: en.json 키 누락 시 vitest 단언 fail.

### 위험 노트 (next session이 빠질 수 있는 함정)

1. **`ChatEvent::Completed`에 `finish_reason` 추가 = 와이어 변경**. 기존 frontend가 강하게 narrow하면 deserialize fail. 반드시 `#[serde(default)]` + IPC 타입에 optional 처리. 기존 ChatPage(`/chat`)도 함께 컴파일 OK 확인.
2. **`chat_stream` 5번째 인자 추가는 모든 호출처 컴파일 영향**. lmstudio adapter / chat IPC / personas 3군데. 한 번에 다 만지지 말고 `Option<&SamplingParams>` 추가 → `None` default로 1차 빌드 → 페르소나 1군데만 `Some` 채택. lmstudio는 v0.8.5에서 시그니처 통일.
3. **chunking 토큰 추정 under-cut 위험**. "1글자 × 2.5 토큰"은 conservative하지만, *주관식 응답에 영문 코드 블록*이 섞이면 영문은 더 짧음 → 추정이 over. *over는 안전 (chunk 더 잘게 나뉨), under만 위험*. under로 안 빠지게 한국어 휴리스틱 보수 유지.
4. **localStorage v1 suffix**. 다음에 schema 바꾸면 v2로 — 기존 v1 키는 무시 (마이그레이션 X 정책 — 사용자가 한 번만 다시 설정).
5. **SamplingDrawer Drawer 컴포넌트가 이미 디자인 시스템에 있는지 확인**. 없으면 inline 작성 — `apps/desktop/src/components/Drawer.tsx` 재사용 또는 `PersonaSurveyDrawer` 로컬 컴포넌트.
6. **a11y 게이트**: SamplingDrawer는 dialog role + focus trap + Esc 필수. CLAUDE.md §4.3.
7. **카피 톤 회귀**: "Inference Parameters" 영어 직역 X. "추론 설정" / "추론 파라미터" 둘 중 하나로 통일. 한국어 i18n 1차.
8. **graceful disconnect vs Length 구분**. ChatOutcome::Completed지만 finish_reason은 다름 — UI 분기 명확히. Aborted는 "잘렸어요" 카피 X (사용자 cancel일 수도).
